## 1. 故障概述

### 1.1 故障现象
- SMB服务无法启动，状态显示为 `failed (Result: exit-code)`
- 系统日志显示错误：`messaging_reinit() failed: NT_STATUS_DISK_FULL`
- 端口445和139未监听
- 服务启动后立即退出，返回状态码1

### 1.2 影响范围
- Windows/Mac/Linux客户端无法通过SMB协议访问服务器共享
- 文件共享服务完全中断
- 可能影响依赖SMB共享的应用程序

## 2. 诊断过程

### 2.1 初步检查
| 检查项 | 命令 | 结果 |
|--------|------|------|
| 服务状态 | `systemctl status smb` | failed (Result: exit-code) |
| 端口监听 | `netstat -tulpn | grep -E "(445|139)"` | 端口未监听 |
| 进程状态 | `ps aux | grep -E "(smb\|samba)"` | 无smb相关进程 |

### 2.2 深入分析
| 分析步骤 | 检查内容 | 发现的问题 |
|----------|----------|------------|
| 日志分析 | `journalctl -u smb -n 30` | 关键错误：`NT_STATUS_DISK_FULL` |
| 磁盘空间 | `df -h /` | 根分区使用率100%，可用空间20K |
| 服务配置 | `cat /etc/samba/smb.conf` | 配置正常，使用默认设置 |
| 系统资源 | `free -m` | 内存充足，无内存压力 |

### 2.3 根因确定
**根本原因**：服务器根分区磁盘空间已满，导致Samba服务初始化失败。

**错误链分析**：
```
磁盘空间不足 (100%使用率)
  ↓
SMB服务初始化需要写入临时文件
  ↓
messaging_reinit() 函数失败
  ↓
返回 NT_STATUS_DISK_FULL 错误
  ↓
smbd进程panic并退出
  ↓
systemd报告服务启动失败
```

## 3. 解决方案

### 3.1 紧急处理步骤

#### 步骤1：清理系统日志
```bash
# 查看日志占用空间
journalctl --disk-usage

# 清理7天前的归档日志
journalctl --vacuum-time=7d
```
**结果**：释放了 **3.3GB** 磁盘空间

#### 步骤2：检查磁盘空间分布
```bash
# 检查各目录占用空间
du -sh /* | sort -hr | head -10

# 检查/var目录占用
du -sh /var/* | sort -hr | head -10
```
**发现主要占用**：
- `/nfs` - 192GB
- `/var/lib/docker` - 117GB
- `/data` - 66GB
- `/root` - 18GB

#### 步骤3：启动SMB服务
```bash
# 尝试启动服务
systemctl start smb

# 检查服务状态
systemctl status smb

# 检查端口监听
netstat -tulpn | grep -E "(445|139)"
```

### 3.2 验证步骤
| 验证项 | 命令 | 预期结果 | 实际结果 |
|--------|------|----------|----------|
| 服务状态 | `systemctl status smb` | active (running) | ✅ 符合 |
| 端口监听 | `netstat -tulpn \| grep -E "(445\|139)"` | 139和445端口监听 | ✅ 符合 |
| 进程状态 | `ps aux \| grep smbd \| grep -v grep` | 有smbd进程运行 | ✅ 符合 |
| 服务日志 | `journalctl -u smb --since "10:34:00"` | 无错误信息 | ✅ 符合 |

### 3.3 长期措施


## 6. 根本原因分析与预防

### 6.1 直接原因
- 根分区磁盘空间耗尽（100%使用率）
- 缺乏磁盘空间监控告警
- 日志管理策略不完善

### 6.2 间接原因
1. **Docker资源管理不当**：
   - 容器日志未限制大小
   - 未使用的镜像和容器未及时清理
   - Docker数据目录未独立分区

2. **存储规划不足**：
   - 根分区容量规划不合理
   - 业务数据与系统数据混合存储
   - 缺乏容量预警机制

3. **运维流程缺失**：
   - 缺乏定期磁盘清理计划
   - 监控告警配置不完善
   - 缺乏容量管理流程

### 6.3 预防措施
| 预防措施 | 执行方法 | 负责人 | 完成时间 |
|----------|----------|--------|----------|
| 磁盘监控告警 | 配置Prometheus磁盘使用率>85%告警 | 运维团队 | 2026-02-09 |
| 定期清理计划 | 每周自动清理Docker未使用资源 | 自动化脚本 | 2026-02-16 |
| 日志管理优化 | 配置日志轮转和自动清理 | 系统管理员 | 2026-02-23 |
